JVM

1.java的内存结构(运行时数据区)

  • 程序计数器：线程私有，记录正在执行的虚拟机字节码指令地址；如果正在执行的是native方法，这个计数器值则为空

  • java虚拟机：线程私有，描述的事java方法执行的内存模型；每个方法在执行的同时创建一个栈帧用于存储局部变量表、操作数栈、动态链接等。每个方法调用直至执行完成的过程，就对应着一个栈帧在虚拟机栈中入栈到出栈的过程。

  • 本地方法栈：线程私有，本地方法栈与虚拟机栈所发挥的作用非常相似，他们之间的区别不过是虚拟机栈为虚拟机执行java方法，而本地方法栈执行的是native方法

  • Java堆：线程共享，用来存放对象实例

  • 方法区：线程共享，用于存放以及被虚拟机加载的类信息、常量、静态常量等数据

  • 运行时常量池：是方法区的一部分，用于存放编译期生成的各种字面量和符号引用

2.什么是双亲委派模型

        如果一个类加载器收到类加载的请求，它首先不会自己去加载这个类，而是把这个请求委派给父类加载器去完成，每个层次的加载器都是如此，因此所有的加载请求最终都应该传到顶层的启动类加载器中，只有父类加载器无法完成这个加载请求的时候，子类加载器才会尝试去加载。

3.java虚拟机有哪些类加载器

  • 启动类加载器(Bootstrap ClassLoader):

    这个类加载器负责加载lib目录下的

  • 扩展类加载器(Extension ClassLoader):

    负责加载lib/ext目录中的

  • 应用程序类加载器(Application ClassLoader):

    负责加载用户类路径上的类库，开发者可以直接使用这个类加载器，如果应用程序没有自定义过类加载器，一般情况下程序默认是应用程序类加载器

  • 自定义类加载器：

    用户自定义的类加载器

4.类加载的过程

类加载的过程包括：加载、验证、准备、解析、初始化，其中验证、准备、解析统称为连接

  • 加载：通过一个类的全限定名来获取定义此类的二进制字节流，在内存中生成代表这个类的java.lang.Class对象

  • 验证：确保Class文件的字节流中包含的信息符合当前虚拟机的要求，并且不会危害虚拟机的安全

  • 准备：为静态遍历分配内存并设置初始值

  • 解析：将常量池内的符号引用替换为直接引用

  • 初始化：到了初始化阶段，才真正开始执行类中的定义的java初始化程序代码

5.为什么使用双亲委派模型

        双亲委派模型保证了Java程序的稳定运行，可以避免类的重复加载（JVM 区分不同类的方式不仅仅根据类名，相同的类文件被不同的类加载器加载产生的是两个不同的类），也保证了 Java 的核心 API 不被篡改。如果没有使用双亲委派模型，而是每个类加载器加载自己的话就会出现一些问题，比如我们编写一个称为 java.lang.Object 类的话，那么程序运行的时候，系统就会出现多个不同的 Object 类。

6.为什么要破坏双亲委派模型

        使用双亲委派模型无法满足需求，以Tomcat为例，一个web容器可能要部署一个或者多个应用程序，不同的应用程序可能会依赖同一个第三方库的不同版本，因此需要保证每一个应用程序的类库都是独立的，相互隔离的

7.如何破坏双亲委派模型

        破坏双亲委派模型的思路类似，以Tomcat为例。

  1. 继承ClassLoader，Tomcat中的WebappClassLoader继承ClassLoader的子类URLClassLoader
  2. 重写loadClass方法，实现自己的逻辑，不需要每次都先委派给父类加载

8.java中有哪几种常量池?

  1. class文件常量池：用于存放编译期生成的各种字面量(相当于常量)和符号引用
  2. 运行时常量池：class文件时类被编译为class文件时生成的，而当类被加载到内存中后，jvm就会将class文件常量池中的内容存放到运行时常量池中
  3. 字符串常量池：用来存放字符串，全局只有一份，被所有类共享

9.String中的intern方法

        在jdk 7之后的版本中，作用为：如果字符串常量池中已经有这个字符串了，则直接返回字符串常量池中的引用。如果没有，则将这个字符串的引用保存一份到字符串常量池，然后返回这个引用。

10.怎么判断对象已经死去？

常见两种判别方法：引用计数法和可达性分析算法，HotSpot中使用可达性分析算法

  • 引用计数法

    给对象添加一个引用计数器，每当有一个地方引用它时，计数器值就加一，当引用失效时，计数器值就减一。任何时刻计数器值为0的对象就不可能就再被使用

    优点：实现简单，判定效率高

    缺点：很难解决对象之间循环引用的问题

  • 可达性分析算法

    通过一系列的称为“GC Roots”的对象作为起始点，从这些节点开始向下搜索，搜索所走过的路径称为引用链，当一个对象到GC Roots没有任何引用连接，则证明次对象时不可用的。

11.哪些对象可以作为GC Roots？

  1. 虚拟机栈中的引用对象

  2. 本地方法栈中的引用对象

  3. 方法区中类静态属性引用对象

  4. 方法区中常量引用对象

12.介绍四种引用

  • 强引用：在代码中普遍存在，只要强引用还存在，垃圾收集器永远不会回收的

  • 软引用：用来描述一些还有用但非必须的对象，使用SoftReference实现，在系统将要放生内存溢出的时候，将会进行回收

  • 弱引用：用来描述非必须的对象，使用WeakReference来实现，在进行垃圾收集的时候将会被回收

  • 虚引用：是最弱的一种引用关系，使用PhantomReference类来实现，虚引用必须和引用队列联合使用，当垃圾回收器准备回收一个对象时，如果发现他还有虚引用，就会在回收对象的内存之前，把这个虚引用加入到与之关联的引用队列中。通过判断队列中是不是有对象来判断对象是不是要回收。

13.垃圾收集有哪些算法

  1. 标记-清除算法

    首先标记出所有需要回收的对象，在标记完成后统一回收所有被标记的对象

    缺点：

    • 效率问题，标记和清除两个过程的效率都不高

    • 空间问题，会产生大量不连续的碎片

  2. 复制算法

    将内存按容量划分为大小相等的两块，每次只使用其中的一块，当这一块内存使用完，就将还或者的对象复制到另一块上面，然后再将已经使用过的内存空间一次清理掉。

  3. 标记-整理算法

    标记阶段与“标记-清除”算法一样，而后续是将所有存活的对象都向一端移动，然后直接清理掉端边界以外的内存

14.HotSpot为什么要分为新生代和老年代

        把java堆分为新生代和老年代，这样就可以根据各个代的特点采用最合适的收集算法

在新生代中，每次都有大量的对象死去，只有少量的存活，一般采用复制算法。老年代中对象存活率高，一般采用标记-清理或者标记-整理算法

15.新生代中Eden区和Survior区的默认比例

新生代分为1个Eden区和2个Survior区，通常称为From Survior和To Survior

Eden区和Survior区默认比例：8：1：1

16.HotSpot GC的分类

分为两大类

  1. Partial GC：不收集整个GC堆

    • Young GC/Minor GC：只收集新生代的GC

    • Old GC：只收集老年代

    • Mixed GC：收集整个新生代以及部分老年代

  2. Full GC/Major GC：收集整个GC堆的模式

    包括新生代、老年代、永久代等

17.GC触发的条件

  1. Young GC：当新生代中的Eden区没有足够的空间进行分配的时候会触发

  2. Full GC：

    • 当准备触发Young GC时，如果发现数据比目前老年代剩余空间大，就不会触发Young GC而是触发Full GC

    • 如果有永久代，在永久代需要分配空间但是已经没有足够的空间时，也要触发一次Full GC

    • System.c()默认也是触发Full GC

18.什么情况下新生代对象会晋升到老年代？

  1. 如果新生代的垃圾收集器为Serial和ParNew，当对象大于设置的参数值时，会被认为时大对象，直接进入老年代

  2. Young GC后，如果对象太大无法进入Survivor区，则会通过分配担保机制进入老年代

  3. 当Survivor区中的对象年龄增加到一定程度(默认是15)，就会被晋升到老年代中

  4. 如果Survivor区中相同年龄所有对象大小的总和大于Survivor空间的一半，年龄大于或者等于该年龄的对象就可以直接进入老年代

19.发生Young GC的时候需要扫描老年代对象吗

不需要扫描，如果扫描的话Young GC的效率会下降

20.垃圾收集器有哪些？

  • Serial收集器：只会用一条垃圾收集线程去完成垃圾收集工作，在进行收集时必须暂停其他线程工作(STW)

    • 新生代采用：标记-复制

    • 老年代采用：标记-整理

    优点：简单而高效

    缺点：会有STW

  • ParNew收集器

    就是多线程版本的Serial收集器

    • 新生代：标记-复制

    • 老年代：标记-整理

  • Parallel Scavenge收集器

    Parallel Scavenge看上去与ParNew一样，但它注重吞吐量，吞吐量是执行用户代码时间与CPU总耗时

    • 新生代：标记-复制

    • 老年代：标记-整理

  • Serial Old收集器

    Serial收集器老年代版本

    用途：

    • JDK 1.5之前与Parallel Scavenge搭配使用

    • 作为CMS的后备方案

  • Parallel Old收集器

    Parallel Scavenge收集器的老年代版本

21.介绍CMS垃圾收集器的特点

        CMS(Concurrent Mark Sweep)收集器是一种以获取最短回收停顿时间为目标的收集器。CMS是HotSpot虚拟机第一款真正意义上的并发收集器。

        CMS是以标记-清除算法实现的，分为四个步骤

  • 初始标记：暂停其他线程，记录与root相连对象，这个阶段速度很快

  • 并发标记：同时开启GC与用户线程，会记录跟踪发生引用更新的地方

  • 重新标记：修正并发标记期间因用户线程运行而产生变动的标记记录，时间比初始标记长，但远比并发标记短

  • 并发清除：开启用户线程，同时GC开始对未标记的清除

优点：并发收集，低停顿

缺点：对CPU资源敏感，利用并发清除算法会产生大量碎片
